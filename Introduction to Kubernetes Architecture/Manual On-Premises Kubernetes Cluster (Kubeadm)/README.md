# Manual On-Premises Kubernetes Cluster (Kubeadm)

**Objective:** Provision a multi-node Kubernetes cluster from scratch using `kubeadm` to understand the underlying architecture of the Control Plane, Worker Nodes, and Container Network Interfaces (CNI).

**Tech Stack:** AWS EC2 (Ubuntu 24.04), `containerd`, `kubeadm`, `kubelet`, `kubectl`, Flannel CNI.

### 1. Infrastructure Provisioning & Security

- Provisioned 3 AWS EC2 instances (`t2.medium`) running Ubuntu: 1 Master Node and 2 Worker Nodes.
- Configured AWS Security Groups to allow internal traffic (`0.0.0.0/0` for lab purposes) so the nodes could communicate over port `6443` (API Server) and overlay network ports.

### 2. Node Preparation (All Nodes)

To ensure Kubernetes had full control over system resources, swap memory was disabled. The kernel was configured to allow IPv4 forwarding and bridge netfilter traffic. `containerd` was installed and configured as the primary container runtime.

### 3. Control Plane Initialization (Master Node)

The cluster brain was bootstrapped using `kubeadm`.

Bash

`kubeadm init --pod-network-cidr=10.244.0.0/16`

Once the API server, etcd, and scheduler were running, the `kubectl` configuration was exported to the user's home directory.

### 4. Container Network Interface (CNI) Setup

To allow Pod-to-Pod communication across different servers, the **Flannel** overlay network was deployed directly to the control plane.

Bash

`kubectl apply -f https://github.com/flannel-io/flannel/releases/latest/download/kube-flannel.yml`

### 5. Worker Node Integration

Using the secure bootstrap token and CA certificate hash generated by the Master node, the worker nodes were successfully joined to the cluster.

### 6. Cluster Verification

Verified the cluster state and node readiness via the Kubernetes CLI.
